좋아. Cursor가 짚은 “보완 포인트”들은 전부 **정확**하고, 지금 시점에서 문서에 **한 줄씩만 못 박아도** 실행 중 혼선을 크게 줄일 수 있어. 아래는 **추천 보완안(=문서에 추가할 내용 + 왜 필요한지 + 어디에 넣을지)**를 우선순위로 정리한 거야. (코드 지시 없음)

## P0 — 문서에 “결정”으로 박아야 하는 6줄 (가장 중요)

### 1) Repo/템플릿 전략을 Phase 0에 확정문으로 박기

**추천 결론(현실적)**

* `eventflow`를 **클론해서** `inev-template`(골든 템플릿) 새 레포로 시작
* 현재 `inev` 레포가 이미 있다면: “기존 inev는 폐기/보관(archive), 새 템플릿을 기준으로 진행”으로 명시

**왜**: 템플릿이 무엇인지 애매하면 이후 인스턴스 생성/전파 전략이 계속 흔들림.

**어디에**: Phase 0 첫 항목 바로 아래에 `Decision: Template Repo` 섹션 3줄 추가.

---

### 2) “Client(폴더)” 정의를 1문장으로 고정

너희 논의상 **서브도메인 앱 1개 = 클라이언트 1개**가 기본이야.

**추천 문장**

* `Client`는 “서브도메인 앱의 소유 조직(1:1)”이며, **해당 앱 내부에서 이벤트를 폴더처럼 묶는 상위 그룹**이다.
* (선택) “향후 한 앱에 여러 Client를 넣는 멀티테넌시는 하지 않는다(또는 Phase 9 이후 검토)”라고 선언

**왜**: UI/스키마/RLS가 전부 이 정의에 종속됨.

**어디에**: 4.3(권한 모델) 또는 6.1(Event 컨테이너) 직전에 “Client 정의” 박스 1개.

---

### 3) DB 스키마 전환 범위를 “신규 스키마”로 확정(리빌드니까)

Cursor 지적처럼 EventFlow 스키마 재사용/신규가 불명확하면 Phase 1부터 막혀.

**추천 결론**

* `inev.ai`는 **신규 스키마**로 간다(권장).
* EventFlow 스키마는 “참고”만 하고, 이벤트 컨테이너 중심(events/leads/participations/modules/logs)으로 최소 모델부터 구축.
* EventFlow의 테이블을 억지로 끌고 오지 않는다(꼬임 재생산 방지).

**왜**: 리빌드의 목적이 “꼬임 방지 구조 재설계”이기 때문.

**어디에**: Phase 1 시작 전에 `Schema Strategy: New` 섹션 추가.

---

### 4) Admin 인증 방식과 공용 Supabase에서의 tenant 구분을 한 줄로

**추천 결론(단순 & 안전)**

* Admin은 Supabase Auth 사용(이메일/비번 또는 매직링크 중 1개 선택)
* 공용 Supabase를 쓸 때는 **client_id를 모든 핵심 테이블의 1급 컬럼으로 두고**, Admin은 자신의 client_id 범위만 보도록 RLS를 건다.

**왜**: “공용 DB + 여러 서브도메인 앱”이면 운영 실수(다른 클라 데이터 노출) 방지장치가 필요.

**어디에**: Phase 0 “키 관리 방식” 아래 `Auth & Tenant Isolation` 5줄.

---

### 5) Novel.sh 이식 문서 위치를 “링크/경로”로 고정

네가 업로드한 가이드가 있으니, 문서에 명시만 하면 됨.

**추천 문장**

* “Novel 이식은 ‘블로그_에디터_구현_완전_가이드.md’를 표준 체크리스트로 사용한다.”

**어디에**: Phase 4 태스크에 “참조 문서”로 1줄.

---

### 6) `/s`와 `/entry` 관계를 한 줄로

**추천 결론**

* `/s/{code}`는 “클릭 로그를 남기고 최종 목적지로 리다이렉트”하는 게이트이며,
  **SMS/외부채널은 최종 목적지를 기본적으로 `/entry`로 둔다.**
* 즉: `/s` → `/entry` → (입장하기 버튼) → 실제 이벤트 페이지

**왜**: Phase 5(Entry Gate)와 Phase 7(Short link) 개발이 겹치면 설계 충돌이 잦음.

**어디에**: 6.7 Short Link 섹션에 2줄 추가.

## P1 — 실행 품질을 올리는 “권장 보완” 5개

### 7) “이벤트 1개만 있는 앱” vs “여러 이벤트” 지원 여부

**추천 결론**

* 기본은 “여러 이벤트 목록”을 지원하되, 설정으로 “단일 이벤트 랜딩(바로 진입)”도 허용

**왜**: 고객마다 운영이 다름(시리즈 vs 단발). Cursor가 지적한 대로 Phase 1에서 결정 필요.

---

### 8) D1~D5 확정 항목을 문서 상단에 “Confirmed Decisions”로 승격

Cursor가 말한 것처럼, 이건 문서 읽는 사람 혼선을 확 줄여.

**추천**

* 문서 2장(전제 조건) 바로 아래에 “✅ Confirmed Decisions” 블록 삽입
* 본문 D1~D5는 “참조”로만.

---

### 9) 중복 로그인 방지: “왜 둘 다 튕기는지”와 “바꿀 목표”를 명확히

이건 이미 너희가 문서화 했고, 현행 구조가 “각자 내가 새 세션이라 가정하고 브로드캐스트”라 경합에 취약해. 

**추천 문장**

* “현행은 타이밍 경합 시 양쪽이 서로에게 conflict를 보내 둘 다 튕길 수 있으므로, 승자 결정 기준(lease)을 단일화한다.” 

---

### 10) Email/메시징 모듈: 예약 발송을 Phase 4에서 어디까지 할지 한 줄

현재는 예약 저장만 있고 실행 스케줄러가 없음.
**추천**: Phase 4는 “예약 저장 + 수동 실행”까지, 자동 스케줄러는 Phase 7/8로.

---

### 11) “로그는 쌓고 분석은 나중”을 CRM 모듈로 명시

지금 아이디어 단계라면, 문서에 “CRM은 Phase 9+”로 분리하고
**Event Log 표준**만 Phase 2~3에서 도입한다고 적으면 됨.

## P2 — Phase 0~2 태스크 보드로 쪼개기(추천 포맷)

Cursor가 제안한 “태스크 보드(체크박스 + 담당 파일/영향 범위/롤백)”는 바로 효과가 커.

**권장 구조(문서 섹션)**

* Phase 0: Repo/DB/Vercel/Env/RLS
* Phase 1: Client/Event/Module 스켈레톤 + 라우팅
* Phase 2: Registration/Survey 최소 완성

각 태스크마다:

* **Outputs**(생기는 파일/테이블)
* **Risks**
* **Rollback**
* **DoD**

이렇게만 써도 Cursor가 실행을 훨씬 안전하게 할 수 있어.

Cursor가 짚은 보완점은 전부 “맞는 방향”이고, **P0의 6개(Repo/Client/Schema/Auth&RLS/Novel 참조/s→entry)**를 문서에 확정문으로 박는 것만 해도 리빌드 실행 난이도가 크게 내려간다.

원하면 내가 **Phase 0~2를 ‘실행용 태스크 보드’ 형태**로 바로 작성해서, Cursor가 그대로 작업하도록 만들어줄게.
